Commit review verdicts must stay hash-scoped. A repository can look healthier at `HEAD` because later commits fixed a problem, but that does not make the earlier commit safe to approve. In this session, portal fixes, auth response changes, and tag-management changes all showed why this matters. The reliable pattern is: inspect the target commit's own diff, compare its behavior against the feature claim, and only then use current files as supporting context. When context pressure is high, a compact checkpoint listing approved and blocked hashes is more valuable than another long narrative, because it preserves exact decisions for the next session.
