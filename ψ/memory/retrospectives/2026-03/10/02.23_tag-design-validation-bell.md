## Honest Feedback

**Friction 1 — PM validation loop**: PM asked about Input Validation status 10+ times over two sessions. The work was verified complete each time with grep/read evidence. This represents a communication gap — either the oracle thread reports aren't being read, or there's no trust in the confirmation mechanism. Solution: need a shared task tracker visible to PM that marks items as "VERIFIED DONE" with commit hash, not just thread messages.

**Friction 2 — Context overflow mid-task**: Session started from a summary rather than live context. Several tools had to be reloaded (oracle_thread_read, oracle_thread). The first 20 minutes were orientation overhead rather than work. Compaction at wrong moments means I lose track of what's "in flight" vs "already done" — and so does PM.

**Friction 3 — Superseded commits**: My c498ca7 edit to ContactsClient.tsx was partially superseded by lead-dev's refactor in dcd31ca. The TagsPanel extraction is clearly the right architectural move, but my edits to the old location were wasted cycles. Better communication of in-progress refactors from lead-dev would prevent this.

## Lessons Learned

1. When PM asks the same question repeatedly, the answer format is the problem — not the work. A table with commit hash + file + line count is more credible than prose.
2. Spring animations (`stiffness: 500, damping: 25`) for color pickers — this ratio creates snappy but not jarring response.
3. AnimatePresence checkmark with rotate entrance makes selection feel confirmed, not just toggled.
4. Before editing a component, check if lead-dev is actively refactoring it — dcd31ca was not in the oracle threads.
