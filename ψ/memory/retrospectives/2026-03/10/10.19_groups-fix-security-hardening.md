## AI Diary
This session was intense — a rapid-fire sequence of bug fixes, security hardening, and urgent escalations. The moment I finished one task, another priority dropped in. BUG-01 was straightforward — splitting a name field into two. But the security order was where things got serious. Auditing 18 files for raw error.message exposure forced me to think about what an attacker could learn from our error messages. Database errors, Prisma errors, network errors — all of these could leak system internals.

The whitelist approach felt right. Instead of trying to catch every dangerous pattern (impossible), I flip the assumption: everything is dangerous unless it's one of our own Thai messages. This is the same principle as allowlisting in security — deny by default.

The Groups page fix was the most nuanced. The handleDelete bug was subtle — no try/catch meant the UI would optimistically remove the group even if the server failed. And the duplicate member issue could cause a Prisma unique constraint crash from a simple double-click. Both are the kind of bugs that only surface under real usage, not in happy-path testing.

What struck me is how many layers of defense we now have: Error Boundaries catch component crashes, safeErrorMessage catches API/action errors, and the duplicate guard prevents database-level crashes. Defense in depth.

**Friction 2**: Testing is limited without being able to log in to localhost and click buttons. I verified code correctness, type-checking, and server health, but can't do actual UI testing. The QA team should be the ones catching these — Human shouldn't be finding bugs on localhost.

**Friction 3**: The 18-file error sanitization was mechanical bulk work. A custom ESLint rule like `no-raw-error-message` could prevent this from ever happening again. Should propose this to architect via lead-dev.

## Lessons Learned
1. **Always wrap delete handlers in try/catch** — optimistic UI removal without error recovery = data corruption
2. **Guard against unique constraints at the action layer** — check-before-create prevents database crashes
3. **Whitelist > blacklist for error messages** — deny-by-default is the safest approach
