## AI Diary
I spent this session acting as a gate rather than a builder, and that shifted the rhythm in a useful way. The commit looked clean at first glance because the new behavior had visible UI polish: the bell dropdown opened, the badge logic was more deliberate, the search box finally pushed users into the messages screen, and there were no obvious console leftovers in the reviewed paths. But the moment I checked the server-side filtering path, the shape of the bug changed. The new shell search suggested a broad global search, while the server query only searched `recipient` and `content`. Then the client layer on the messages page filtered `senderName` too. That mismatch matters because it creates a false promise: a sender-name search can look supported in the UI and still fail when it originates from the shell. I felt the familiar tension between "this is close enough to ship" and "this will come back as a user-facing defect." The right answer here was to stay strict. I also noticed that the notification routes were materially safer than the search path: both require a valid session and scope data to `user.id`, which is the kind of boring correctness I want to see more often. This was a useful reminder that reviews are rarely all-good or all-bad; they are usually uneven, and the job is to separate a working-looking change from a deploy-safe one.

## Honest Feedback
The first friction point was review target drift. The repo had moved quickly, and the current worktree was not perfectly aligned with the commit under review, so I had to keep separating "what is true in commit `362fb16`" from "what is true in the repo right now." That costs attention and makes it easier to misattribute a bug to the wrong change. The second friction point was duplicated search logic across server and client. Once filtering rules live in two places, correctness becomes fragile. The shell search, server `getMessages()`, and client `MessagesClient` each carried a slightly different definition of what "search" means. That is exactly the kind of inconsistency that creates regressions with no syntax errors. The third friction point was session wrap-up tooling. The `/rrr` skill expects an Oracle sync step, but the local environment did not expose a usable `oracle_learn` command, so I could only complete the vault-writing part. That is survivable, but it means the process is only partially automated and depends on memory at the worst possible time, right when context is already tight.

## Lessons Learned
- A search feature is only correct when server-side and client-side semantics match exactly.
- Session-auth notification routes are fine as long as they scope queries by `user.id`, but UI behavior still needs review separate from auth.
- Commit review in a dirty or drifting worktree must explicitly distinguish commit facts from uncommitted facts.
