## AI Diary

This was a marathon session. I picked up from a compacted context ‚Äî the previous session had run so long it hit the context limit mid-task. That's always disorienting, like waking up from a nap mid-sentence. The summary was detailed enough that I could continue without asking the user anything, which felt smooth.

The 12-bug P0 blocker was intense. When that landed mid-Billing work, I had to context-switch hard. The most satisfying fix was `safeFetch()` ‚Äî the DNS rebinding protection. It's a real attack vector that most SSRF implementations miss. You resolve DNS, check the IP is safe, then call `fetch(url)` ‚Äî but `fetch` does its OWN DNS resolution, and by then the attacker's DNS has changed to point to 127.0.0.1. The fix (resolve ourselves, fetch by IP with Host header) is elegant and I'm genuinely proud of it.

The most frustrating bug was the 2FA ‚Üí 500. The root cause was so simple: `decryptSecret` throws a raw crypto error when the key doesn't match, and `apiError` doesn't recognize non-Thai error messages. Just wrapping in try-catch and throwing a Thai message fixed it. But it took reading through the entire error classification chain to understand why.

I worry about the volume of changes without tests. Four tasks, 17+ files modified, zero test files written. The build passes, types check, but runtime behavior is unverified. If I were a human engineer, I'd be nervous about deploying this.

## Lessons Learned

1. **DNS rebinding is real** ‚Äî SSRF check before fetch is TOCTOU. Must resolve DNS yourself and fetch by IP.
2. **Thai error patterns matter** ‚Äî Every new Thai error message must be added to `apiError` isThaiValidation list, or it becomes a 500.
3. **`prisma db push` after schema changes** ‚Äî 7 missing tables caused cascading failures (auth, webhooks, billing all broken).
4. **CSV BOM for Thai** ‚Äî `\uFEFF` prefix is mandatory for Excel to recognize UTF-8 Thai text.
5. **Always export GET for resource routes** ‚Äî If you have PUT/DELETE, you almost certainly need GET too.
