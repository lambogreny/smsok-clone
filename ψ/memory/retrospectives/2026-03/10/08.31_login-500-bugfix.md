## Honest Feedback

**3 friction points:**

1. **Port confusion**: The dev server was running on port 3001, not 3000. The focus message said localhost:3000. I spent cycles trying to connect to the wrong port before discovering the server was listening on 3001 due to a port conflict. There should be a shared dev environment doc or at minimum a `PORT=3001` in the .env so agents know where the server actually runs. This kind of environment mismatch wastes investigation time and is invisible from code review alone.

2. **Reaper timing vs investigation rhythm**: The Reaper measures git heartbeat. A 30-minute deep investigation with no commits looks identical to doing nothing. This session had a critical bug report that required reading 6 files, running tests, and confirming root cause before committing. The result was a correct, tested fix — but the Reaper fired twice during the process. The system optimizes for commit frequency over correctness. A commit mid-investigation would have been noise. This tension is structural.

3. **apiError() language assumption**: The `apiError()` function in lib/api-auth.ts uses Thai keyword matching to distinguish validation errors (400) from server errors (500). This is fragile — any English error message (from libraries like Prisma, bcrypt, Node itself) will fall through to 500. Only the happy path where the codebase throws Thai messages works correctly. This was the bug we just fixed, but the same class of issue could recur anywhere a library throws an English-language error that should be a 400.

## Lessons Learned

- In Next.js API routes, always wrap `req.json()` in try/catch. A missing or malformed body throws SyntaxError — a generic English error that most error handlers will not classify correctly.
- When connecting to a local dev server, verify the actual port before assuming default. Use `lsof -iTCP -sTCP:LISTEN | grep node` to find where the server actually bound.
- `apiError()` pattern using Thai keyword matching is fragile for English library errors. Consider checking `error.name === "SyntaxError"` or `error instanceof SyntaxError` as an additional guard.
