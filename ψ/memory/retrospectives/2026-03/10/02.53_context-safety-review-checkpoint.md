## AI Diary
I spent this session acting as a gate rather than a builder, and that changed the texture of the work. The pressure was not technical complexity alone; it was the constant requirement to preserve exact verdicts while the session was repeatedly interrupted by urgency and context warnings. I had to keep remembering that a commit review is not a repo review. A later fix can make the current tree look clean, but that does not retroactively make an earlier commit shippable. That distinction mattered several times today, especially around the tags work and the portal fixes. I found myself repeatedly checking whether I was judging a hash by its own diff or by what came after it.

The strongest part of the session was that I stayed anchored to concrete commit boundaries. That prevented false approvals. The weakest part was friction from carrying too much review state in memory across loops. The repo also had local modifications present, which added another layer of caution because I could not casually treat current files as the exact state of every historical commit. Even so, the queue is now much clearer: login route plus fetch-based login are in a good state, while auth reset, tag consistency, and some portal/search changes still need correction. This checkpoint matters because losing the thread here would make the next review cycle slower and less trustworthy.

## Honest Feedback
Three friction points stood out clearly. First, the review loop kept mixing "current repo health" with "historical commit safety." That is dangerous because it invites accidental approval of a broken intermediate commit just because a later commit fixed it. I had to keep actively pushing back against that drift. Second, the repo had local uncommitted changes in important files such as auth and tags. That made historical inspection noisier and forced extra caution when mapping issues back to specific hashes. Third, repeated urgency prompts increased the risk of shallow summaries. Under pressure, the easiest failure mode is to answer fast with a vague verdict instead of preserving the exact reason a commit is blocked. I avoided that, but it took deliberate effort. The next session should keep using short, hash-specific verdicts and avoid broad "looks good now" language unless the user explicitly asks for a current-tree review rather than commit review.

## Lessons Learned
- Commit review must stay hash-scoped. Later fixes do not rescue earlier broken commits.
- Tags remain a structural risk area because UI and DB models are not fully unified.
- Auth hardening needs to be checked at the response-shape level, not just message text.
- Under context pressure, the highest-value artifact is a short checkpoint with exact approved and blocked hashes.
