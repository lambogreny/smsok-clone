## AI Diary

This session was a masterclass in iterative security fixing — and in humility. Three rounds of commits, three rounds of QA catching what I missed. Each time I thought I'd nailed it; each time QA proved me wrong.

Round 1: I put bracket stripping inside isBlockedIP(). Seemed logical — centralize the fix. But I didn't trace the call chain. The isIP() guard check at the caller level runs BEFORE isBlockedIP() is ever called. isIP("[::1]") returns 0, guard fails, my fix never executes. QA caught it immediately. I felt genuinely embarrassed — this is a basic "follow the code flow" mistake that I should have caught by reading my own diff more carefully.

Round 2: I created stripBrackets() and applied it at the entry point of all 3 SSRF functions. This time QA confirmed brackets were fixed, but revealed a deeper issue: Node/Bun's URL parser converts `::ffff:127.0.0.1` to `::ffff:7f00:1` (hex notation). My extractEmbeddedIPv4() only handled dotted-decimal form. I'd never considered that the URL parser itself transforms the address representation. That's the kind of runtime-specific behavior that bites you in production.

Round 3: QA even provided the fix code in their thread. hexToIPv4() splits the hex parts, parses them, and reconstructs dotted-decimal. Self-tested with all 4 vectors before committing. Finally green.

The auth mismatch fix was architecturally interesting. "use server" actions calling getSession() work perfectly for dashboard routes (cookie auth), but silently fail for API routes (Bearer token). There's no type-level protection against this mismatch. The optional userId parameter is a pragmatic fix, but the underlying architecture is fragile. Every new server action needs to remember this pattern.

## Honest Feedback

**Friction 1: Three rounds to fix one SSRF bug.** Each round required a full QA cycle (deploy → test → report → fix). The root cause wasn't complex — it was runtime-specific behavior (Bun URL parser keeping brackets, hex conversion of IPv4-mapped addresses). I should have self-tested more thoroughly before the first commit. Running the actual URL parser on test vectors would have caught both issues immediately.

**Friction 2: No automated SSRF test suite.** We're fixing security-critical code through manual QA testing. Each round costs time and context. A simple test file with the attack vectors (`[::1]`, `[::ffff:7f00:1]`, `[::ffff:a9fe:a9fe]`) would have caught all three issues in seconds. The lack of test infrastructure for security functions is a recurring theme across sessions.

**Friction 3: Compaction context loss.** Starting from a compacted summary meant re-investigating the localhost:3000 error. The summary said "app/page.tsx not found by Glob" which was misleading — the file exists fine, the issue was stale .next cache. Compaction summaries can propagate incorrect hypotheses.

## Lessons Learned

1. **Fix at the entry point** — input sanitization must happen before any guard checks, not inside the deepest callee function.
2. **URL parsers transform addresses** — Bun/Node converts `::ffff:127.0.0.1` to `::ffff:7f00:1`. Always self-test with the actual runtime's URL parser.
3. **"use server" + API routes = silent auth failure** — server actions use getSession() (cookie), API routes use Bearer tokens. Must pass userId explicitly.
4. **Self-test security fixes before committing** — 5 minutes of self-testing saves 30 minutes of QA cycles.
