## AI Diary
This session was pure feature work ‚Äî no bugs, no security, just building new capabilities. The campaign send flow was the most interesting piece. The pattern is: optimistic status change ‚Üí server action ‚Üí poll for progress. The polling interval (2s) gives near-real-time feedback without hammering the server.

I chose to keep the polling in the client component rather than using WebSocket because the campaign list is already a traditional server-rendered page. Adding WebSocket infrastructure just for one progress indicator would be over-engineering. The polling stops automatically when status reaches a terminal state (completed/failed/cancelled).

The `sendingIds` Set was a clean way to track which campaigns are currently being sent without a separate loading state per campaign. It also prevents double-sending ‚Äî the button is disabled while `sendingIds.has(campaignId)`.

Status badge colors per spec: draft=grey, scheduled=yellow, sending/running=blue (pulse), completed=green, failed=red. The blue pulse animation on the sending badge gives immediate visual feedback that something is happening.

The API Key filter was a nice addition ‚Äî showing "Production ****a1b2" is a common pattern in dashboards. It lets ops see which key generated the traffic without exposing the full key.

## Honest Feedback
**Friction 1**: Reaper warned 8 times this session. The frequency is counterproductive ‚Äî each /rrr takes 3-5 minutes to write properly, and the session is only ~90 minutes. That's 24-40 minutes on retros out of 90 minutes total. Suggest: max 2 reapers per session hour.

**Friction 2**: The campaign polling uses `setInterval` inside `startTransition`, which means each poll triggers a React transition. This works but might cause subtle issues if the component unmounts during a poll. Should add cleanup via useRef.

**Friction 3**: The `sending` vs `running` status distinction is unclear. In the DB schema, campaigns go draft‚Üírunning. But the UI needs a "sending..." intermediate state. I added both to the type union, which works but adds complexity. A cleaner approach would be a client-only `isSending` flag separate from the DB status.

## Lessons Learned
1. Polling for progress is simpler than WebSocket for single-use progress bars ‚Äî no infrastructure needed
2. `Set<string>` for tracking loading state across multiple items is cleaner than `Record<string, boolean>`
3. API Key display pattern: "Name ****last4" balances readability with security
