## AI Diary

This session was a lesson in defense-in-depth and the importance of understanding execution flow. My first SSRF bracket fix was technically correct — isBlockedIP() *does* strip brackets now — but it didn't matter because the guard check (`isIP(hostname)`) runs first and returns 0 for bracketed addresses, so the code never even reaches isBlockedIP(). QA caught this perfectly: "fix put bracket stripping in isBlockedIP() but the GUARD CHECK runs FIRST."

The auth mismatch bug (C2) was a classic "use server" / API route architecture clash. Next.js server actions use getSession() (cookie-based), but API routes authenticate via Bearer token (authenticateApiKey). The route called authenticateApiKey, got the user, then called the server action which internally called getSession() — ignoring the API key entirely. The fix was straightforward: add optional userId params and fall back to getSession() when not provided. This preserves both code paths (dashboard cookies + API keys).

I'm proud of catching and fixing the "ต้อง" pattern issue proactively. Every Thai error message with "ต้อง" (must/should) was incorrectly classified as 500 Internal Server Error. Adding it to the isThaiValidation list turns them all into proper 400 Bad Request responses.

The JSON parse fix on SMS send was a one-liner in spirit — just wrap req.json() in try-catch. But it represents an inconsistency: some routes handle malformed JSON gracefully, others don't. A middleware-level solution would be better long-term.

## Honest Feedback

**Friction 1: First fix was wrong.** I put bracket stripping inside isBlockedIP() without tracing the call chain. The isIP() guard check at the caller level runs first and silently fails for bracketed addresses. I should have read the code flow more carefully before committing. QA had to catch my mistake.

**Friction 2: "use server" + API route architecture is confusing.** Server actions with getSession() work for dashboard routes (cookie auth), but silently fail for API routes (Bearer token auth). There's no type-level enforcement — nothing stops you from calling a getSession()-based action from an API route. This will keep causing bugs until there's a clearer separation.

**Friction 3: Compaction context loss.** Starting from compacted context meant I had to re-investigate the localhost:3000 error from scratch. The summary mentioned "app/page.tsx not found" which turned out to be a Glob search issue, not a real missing file. Cost ~5 minutes of false investigation.

## Lessons Learned

1. **Fix at the entry point, not the callee** — if a guard check (isIP) runs before your fix (bracket strip in isBlockedIP), the fix never executes. Always trace the full call chain.
2. **"use server" actions + API routes** — server actions call getSession() internally. API routes must pass userId explicitly or the action won't know who the user is.
3. **Thai error keyword "ต้อง"** — must be in isThaiValidation or all "must/should" errors become 500.
