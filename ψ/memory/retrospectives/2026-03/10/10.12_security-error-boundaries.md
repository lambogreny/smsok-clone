## AI Diary
This session felt like being a security inspector going through every room of a building, checking every lock and window. When the audit revealed 18 files with raw error.message exposure, I understood the scope immediately — this wasn't just a cosmetic fix, it was a real security concern. Internal error messages can leak database structure, file paths, library versions — information an attacker shouldn't see.

The approach I chose was pragmatic: a whitelist-based sanitizer. Rather than trying to classify what's dangerous (impossible to be exhaustive), I flip the logic — only known-safe Thai messages from our own server actions pass through. Everything else gets a generic message. This is defense-in-depth thinking: even if a new error type appears tomorrow, users will never see raw internals.

The Error Boundary components were satisfying to build. They match the Web3 dark theme perfectly — glassmorphism cards with animated entry, clear CTAs. When a page crashes, users see something beautiful and helpful instead of a blank screen or React error overlay.

What I find interesting about this session is the layering: validation-level fixes (register schema), data-level fixes (groups limit), UI enhancements (SMS history), and security hardening (error boundaries + sanitization) — all in one focused burst. The codebase is measurably more secure now.

## Honest Feedback
**Friction 1**: The audit of 18 files was mechanical but necessary. Having to read imports, add the safeErrorMessage import, then replace patterns file-by-file is tedious. A codemod tool or a project-wide lint rule (e.g., "no raw error.message in catch blocks") would prevent this from recurring. We should consider adding an ESLint custom rule.

**Friction 2**: The registerSchema change cascaded into 3 files (validations, actions, tests) because the schema is used both for form validation and server-side parsing. The tight coupling means any schema change requires touching multiple layers. Not necessarily bad, but worth noting for future schema changes.

**Friction 3**: The groups page had `limit: 1000` which was clearly a "get all contacts" workaround. Now capped at 100, users with >100 contacts won't see all of them in the group member selector. This needs a proper search/pagination solution in the GroupsPageClient — flagging for next session.

## Lessons Learned
1. **Whitelist > Blacklist for error sanitization** — Only allow known-safe messages through, block everything else
2. **Next.js error.tsx is the right pattern** — Route-segment error boundaries catch all errors in that segment automatically
3. **Schema changes cascade** — registerSchema change touched validations, actions, and tests (3 layers)
