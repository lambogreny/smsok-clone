Missing interface contracts create integration bugs like this.

3. **P0 bug survived to production**: The double-verify was shipped in the original auth flow. No test caught it because there was no integration test for the full register→OTP→register flow. ## Lessons Learned

- When a server action (registerWithOtp) internally calls a verification function (verifyOtpForRegister), the calling frontend MUST NOT also call that function. Verify ownership belongs to one layer.
- OTP "already used" on first attempt always means double-verification, not race condition. Check call chain before assuming concurrency.
