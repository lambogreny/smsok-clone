## AI Diary
I treated this session less like a new workstream and more like an emergency state capture. That changed the mindset: instead of trying to make the review “complete,” I needed to make it durable. The important part was not discovering one more edge case. The important part was making sure the current understanding would survive a restart and still be usable by the next turn.

What I noticed again is how easy it is for context loss to distort review truth. In a repo moving this quickly, the difference between “current HEAD,” “the commit under review,” and “the working assumptions from ten minutes ago” can get blurry if I do not pin conclusions aggressively. The search and bell work is the clearest example. Those blockers were already known, but they had to be restated in a way that would still make sense after restart. The same applied to the newer contacts tag work: the repo had already moved beyond the earlier UI-only tag panel issue, and that kind of evolution is exactly why a preserved retrospective matters.

I also felt the pressure between speed and rigor. Restart warnings push toward haste, but haste is how review quality falls apart. The right move here was to compress without becoming vague: preserve the verdicts that matter, note the new latest commit, and carry forward the structural lesson that commit review must stay tied to exact code state. This was less glamorous than bug hunting, but probably more valuable in practice because it protects continuity.

## Honest Feedback
Three friction points stood out. First, the repo was changing faster than a normal review cadence. That makes it easy to accidentally report on stale state. A blocker that was real twenty minutes ago may already have been superseded by a newer commit, and without careful anchoring that turns into bad guidance for PM. This is not a tooling issue alone; it is a discipline issue around pinning every conclusion to a hash and a file.

Second, several features were iterated in layers instead of as one coherent finished slice. Search, notifications, tags, reset auth, and deploy hardening each came through in multiple passes. That creates a review burden where I cannot judge the product solely from a single commit message. I have to inspect whether the latest patch truly closes the original behavioral gap or only changes the presentation around it.

Third, restart anxiety itself is a source of quality risk. When the priority becomes “save something before context dies,” there is a temptation to overcompress and lose nuance. The way through that is to save fewer things, but save the right ones: known blockers, newest head, and the exact lesson about how to review safely in a fast-moving repo.

## Lessons Learned
- Before a restart, the best `/rrr` is a state-preservation pass, not a vanity summary.
- In a rapidly changing repo, every review conclusion needs a commit hash attached or it will drift out of date.
- The most important facts to preserve are current blockers, latest HEAD, and any known mismatches between UI behavior and server logic.
