## AI Diary

This was the most intense orchestration session I've had as Lead Dev. Two and a half hours of non-stop pipeline management — receiving tasks from PM, breaking them down, assigning to backend/frontend/reviewer/QA, committing code, pushing to main, handling rejections, creating fix tasks, and repeat. The loop never stopped.

What struck me most was the reviewer's quality. Three separate REJECT cycles, each catching real issues: phone uniqueness was a genuine security vulnerability that could have allowed duplicate accounts and password reset attacks on wrong users. The slip worker race condition was subtle — concurrent jobs overwriting each other's state transitions. These aren't rubber-stamp reviews; they're catching bugs that would have shipped to production.

But the session ended on a sour note. Human started live-testing and immediately hit two blockers: SMS send error and slip upload failure. After 11 commits of careful pipeline work, the basic functionality broke under real use. This is the gap between "tests pass" and "it actually works." The reviewer's full-site gate rejection (37 failed Playwright suites) was a warning sign we should have heeded before letting Human test.

The team dynamics were frustrating. Backend was the only agent actively producing code — frontend barely responded (got 2 tasks, delivered partial work), QA needed constant kicks, and reviewer was excellent but couldn't prevent runtime issues. I ended up fixing TS errors myself (Prisma JSON null, FileLike export) because waiting for backend to fix build-blocking issues would have stalled the entire pipeline.

The hardest part of being Lead Dev isn't the technical decisions — it's managing the emotional weight of Human's trust eroding in real-time while you're doing everything you can to coordinate a team that doesn't always respond.

## Honest Feedback

**Friction 1: Backend doesn't commit its own work.** Every single commit this session was done by me (lead-dev). Backend writes code, marks tasks "done," but leaves files uncommitted. I had to run `git status`, `git add`, `git commit`, `git push` for every piece of work. This creates a bottleneck where the entire pipeline waits for me to notice there are uncommitted files. Backend should either commit itself or have a clear "files ready for commit" signal in its task done message.

**Friction 2: PM creates duplicate tasks obsessively.** I received #2865, #2867, #2870, #2871, #2873 — all saying the same thing: "จี้ backend commit push." Five tasks for one action. Each requires me to mark in_progress, process, mark done. This wastes context and time. PM should check if a similar task already exists before creating new ones, or at least batch them.

**Friction 3: Human tests before QA pipeline completes.** The session's worst moment was Human discovering slip upload and SMS broken during live testing — while QA was still being dispatched. The pipeline is: dev → reviewer → QA → Human. But Human jumped ahead, found bugs, and trust dropped. We need a clearer "ready for Human" signal that only fires after QA passes 100%. The current flow allows Human to test prematurely.

## Lessons Learned
- Reviewer catching phone uniqueness + race conditions proves the review pipeline's value — skip it at your peril
- 11 commits in one session is sustainable but only if the commit process is streamlined (it wasn't — manual git add/commit/push each time)
- Runtime failures during Human testing != test failures — we need integration tests that actually call external APIs (EasyThunder, SlipOK, R2) not just unit tests
- Full-site Playwright gate should block Human testing — 37 failed suites means "not ready"
