## Honest Feedback

**Friction 1: Backend completely unresponsive.** 4 rounds of `maw hey backend` with zero response. No commits, no code changes, no status updates. PM had to escalate and eventually authorize lead-dev to fix things directly. The task queue shows 3 tasks in_progress for backend with no progress. If agents can't reliably respond, the orchestration model breaks down — lead-dev becomes a solo developer with extra overhead.

**Friction 2: OOM kills on git push.** The pre-push hook running `next build` (265 pages, Turbopack) requires substantial memory. With multiple Claude agents running in parallel tmux panes, the system runs out of memory and SIGKILL's the build process. Three failed push attempts wasted ~10 minutes. Options: reduce concurrent agents during push, or make the pre-push hook lighter (skip full build if pre-commit already passed).

**Friction 3: Stale tests accumulate faster than they're fixed.** 40 vitest failures remain — all structural mocks testing internal implementation details (function signatures, module structure). Every time code evolves, these tests break not because functionality is wrong, but because they assert implementation details. The test architecture needs rethinking: integration tests > structural mocks. Nobody owns cleaning these up.

## Lessons Learned
- SMS "EasySlip" hardcode was a data-code mismatch — always check what's actually in the DB, not what the code assumes
- Union type narrowing in TypeScript: use early return pattern instead of ternary when branches return different shaped objects
- `Prisma.delete()` throws on missing records — always use `deleteMany()` for defensive deletes
- Pre-push builds can OOM with concurrent agents — consider lighter hooks or sequential agent execution during push
