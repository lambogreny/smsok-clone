## AI Diary

This session felt like triage duty in an ER. The P0 came in hot — "10 pages 404!" — and my first instinct was to look for missing files. I spent significant time proving that all 85 pages exist, all links resolve, TypeScript compiles clean, and the build passes. That's the frustrating part of this kind of bug: everything looks correct statically. The crash only happens at runtime when Prisma's `findUniqueOrThrow` encounters an edge case.

The actual fix was surgical — three lines changed in two files to swap `findUniqueOrThrow` for `findUnique` plus null checks. But the cascading fix across 10 pages was methodical: each server component page needed a try/catch wrapping its data fetching, following the pattern that only the campaigns page had gotten right. I appreciated that the codebase already had an `ErrorState` component ready to go — someone had anticipated this need but hadn't wired it everywhere.

The consent task was a pleasant surprise — already done. The register page had the full Stripe pattern with explicit checkboxes, required/optional separation, and backend logging. Sometimes the best outcome is "no work needed."

The PDPA privacy settings page was the most satisfying fix. Four broken `/legal/*` links pointing to non-existent routes, and a completely static consent history that showed nothing. Wiring it to the real consent APIs transformed it from a mockup into a functional page. Small changes, big impact.

## Honest Feedback

**Friction 1: Investigation overhead.** I spent roughly 40% of the session proving the 404s weren't from missing files. The task description said "10 pages 404" but the real issue was runtime Prisma crashes. Better error reporting from the QA team (error type, stack traces, reproduction steps) would have cut investigation time in half. The gap between "user sees blank page" and "Prisma findUniqueOrThrow uncaught" is wide.

**Friction 2: Inconsistent error handling patterns.** The campaigns page had proper try/catch while 10 other pages didn't. This is a codebase consistency issue — when one page gets a pattern right, there should be a lint rule or code review checkpoint ensuring all similar pages follow suit. The fact that this became a P0 means the gap wasn't caught during development.

**Friction 3: Spec files referenced but missing.** Task #2764 referenced `ψ/specs/SMSOK-REGISTER-CONSENT-V2-PREMIUM-SPEC.md` which doesn't exist. The user also referenced `oracle_task_list()` which isn't a real system function in this context. Communication gaps between task management and actual codebase state waste cycles.

## Lessons Learned

- `findUniqueOrThrow` should almost never be used in server component data fetching — prefer `findUnique` + explicit null handling for graceful degradation
- One page with correct error handling doesn't protect the other 10 — patterns need to be enforced systematically
- Static mockup pages (hardcoded data, no API calls) are technical debt that QA will eventually surface

## Next Steps

- Consider adding an ESLint rule to flag `findUniqueOrThrow` in server action files
