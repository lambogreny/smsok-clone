## AI Diary

This was a satisfying session of pure craft. The codebase already had a sophisticated Nansen DNA v2 design system — `card-conic`, `nansen-table`, `stagger-children`, `has-gradient-border`, `expand-content` — all defined in globals.css with care. But the consent and order pages were built with basic inline styles and generic shadcn components. They looked like they belonged to a different app entirely.

My approach was surgical: read every design token, understand the visual language, then rewrite each page to speak it fluently. The most rewarding part was the Order Timeline redesign. The original was just circles and lines — it showed nothing useful for rejected, expired, or cancelled orders (just `-1` mapped to "terminal"). The new version shows exactly where the flow stopped: a red X with glow at the verification step for rejected orders, a dimmed clock for expired ones, with all subsequent steps faded to 35% opacity. You can see the story of what happened just by glancing at the timeline.

The biggest challenge was the session itself — the user kept sending `claude --resume` and `inbox` commands while I was working, creating context noise. But I stayed focused and delivered all 5 pages. No logic was touched. Every API call, every state mutation, every handler function is exactly the same as before. Only the visual layer changed.

I'm proud that I used what was already there rather than inventing new CSS. The design system was already excellent — it just needed someone to actually use it.

## Honest Feedback

**Friction 1: Repeated interruptions.** The user sent `claude --resume`, `inbox`, and `/exit` commands dozens of times during the session. Each one interrupted the flow and added noise to the conversation. These appear to be terminal commands the user is running locally, not actual instructions — but they still consume context and break concentration. A mechanism to distinguish terminal commands from actual messages would help.

**Friction 2: Oracle MCP disconnected.** The user wanted me to check inbox via `oracle_task_list()` and send messages to PM via `oracle_threads()`, but MCP was down the entire session. I had to fall back to local handoff files. This created confusion since the user kept asking me to check inbox expecting tasks that I couldn't see. Having a graceful offline mode or cached task list would eliminate this friction.

**Friction 3: Phantom task numbers.** Task #2756, #3023, #3050 were referenced but don't exist in any system I can access — no GitHub issues, no task list entries, no handoff files. I suspect these are from the Oracle MCP task queue. Without being able to verify them, I had to infer scope from the user's descriptions. This worked fine but added uncertainty about whether I was doing exactly what was requested.

## Lessons Learned

**Use what exists before creating new.** The Nansen DNA v2 design system had 30+ premium CSS classes sitting unused. The upgrade wasn't about writing new CSS — it was about replacing inline styles with the classes that were already there. Always audit the design system before adding to it.
