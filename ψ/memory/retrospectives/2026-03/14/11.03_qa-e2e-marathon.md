## Honest Feedback

**Friction 1: Inbox polling loop.** The session included 50+ "inbox" messages with no new tasks. Each required me to check task state, confirm nothing changed, and respond. This consumed significant context window and attention without productive output. A mechanism to distinguish "genuine new task" from "heartbeat check" would dramatically improve efficiency. Even a simple convention like "inbox — new task attached" vs "inbox — status check" would help.

**Friction 2: Task ID confusion across sessions.** Tasks #2773, #2800, #2856, #2883, #2888, #2893, #2919 were referenced repeatedly as "pending" even after completion. Session boundaries reset the task list, so each new session genuinely couldn't verify completion. The user (or orchestration system) kept asserting tasks were pending when they weren't. A persistent task store that survives session resets would eliminate this entire class of confusion.

**Friction 3: Dev server instability during E2E.** Turbopack crashes are a known QA blocker documented since the original QA-REPORT.md. Running Playwright against `next dev` will always be fragile for long test suites. The team should consider testing against a production build (`next build && next start`) for E2E reliability. This would likely push pass rates from 93% to 99%+.

## Lessons Learned

1. Run Playwright E2E suites sequentially (1 at a time) against Turbopack dev server to avoid crashes
2. QA role means TEST only — don't fix bugs, report PASS/FAIL with evidence
3. When tests fail en masse (39/53), check for stale selectors before assuming real bugs
