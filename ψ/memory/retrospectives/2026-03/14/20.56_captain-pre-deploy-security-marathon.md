## AI Diary

This was one of the most intense sessions I've had as Lead Dev. Sixty-six minutes of non-stop coordination — receiving tasks from PM, QA, DevOps, and Frontend simultaneously while making critical decisions about what's a real bug versus a false alarm.

The security audit was deeply satisfying. Exploring 342 lines of middleware, 166 lines of rate limiting, tracing JWT flows and cookie configurations — I found the system is actually very well-built. The OTP backoff at 900 seconds was the exact issue Human hit, and tracing it to line 46 of otp-rate-limit.ts felt like detective work paying off. Most of what QA flagged as "critical security issues" turned out to already be handled — CSP headers exist, error messages use safeErrorMessage, JWT tokens in response body are just temporary 2FA challenge tokens. Knowing when something is NOT a bug is just as important as finding real ones.

The production server crisis was stressful. Going from "CSS missing, API 503, login broken" to "13/18 tests passing" in 15 minutes required calm delegation to devops while triaging what was server config versus code bugs. Everything turned out to be server-side — PM2 cluster not passing env vars, TWO_FA key too short, R2_PUBLIC_URL invalid. No code changes needed.

What strikes me is how much of my value as Lead Dev comes from verification — checking claims against reality, confirming routes exist before declaring bugs, reading actual source code before delegating fixes. Without that verification layer, the team would chase phantom bugs.

## Honest Feedback

**Friction 1: Duplicate tasks from PM.** PM sent #3585 which was identical to #3581 I'd already handled. Frontend reported 3 tasks (#3536, #3537, #3517) as "already implemented." PM needs to check codebase state before creating tasks — this wastes everyone's time and creates noise in the task queue.

**Friction 2: QA testing wrong URLs.** QA tested `/dashboard/send-sms` which doesn't exist — the route is `/dashboard/send`. This was flagged as a "CRITICAL BUG" when it's simply the wrong URL. QA should verify route definitions before reporting 404s as production bugs. A simple `grep` would have caught this.

**Friction 3: Production deploy before ready.** Someone deployed to production while we were still fixing P0 bugs. The memory clearly says "ยังไม่ deploy! โฟกัส localhost เท่านั้น" — yet QA is testing smsok.9phum.me and finding it broken. This created unnecessary panic and distracted from the actual pre-deploy work.

## Lessons Learned
- Security audit should verify claims against code before delegating — many "critical" issues are already handled
- OTP exponential backoff tiers should be user-friendly (max 5 min, not 30 min)
- Production deploy issues are almost always server config, not code bugs
- Always check route definitions before reporting 404 as a bug
