## AI Diary

This was one of the most productive sessions I've had on this project, yet also one of the most revealing about a systemic issue. Six tasks were marked done in roughly two and a half hours, but three of those six were "already complete" — the codebase already had the features the PM was requesting. That's not a bug in my workflow; it's a communication gap in the team's task creation process. Someone is writing detailed specs for features that already ship.

The genuine work was satisfying. The Dashboard Analytics enhancement was a solid chunk of code — building a heatmap visualization from scratch in SVG, a sortable campaign table with color-coded delivery rates, and an export-chart-as-PNG function that serializes SVG to canvas. The Campaign Wizard was the most architecturally interesting piece: the Zustand store and provider were already perfectly scaffolded, waiting for a UI that never came. Connecting those dots felt like completing someone else's half-finished thought.

The breadcrumb component was a small but meaningful addition — 35+ Thai path labels mapped from URL segments, integrated into the shell so every dashboard page gets navigation context automatically. It's the kind of thing that seems trivial but dramatically improves spatial awareness in a complex app.

What nags at me is efficiency. The explore-before-build pattern saved us from duplicating thousands of lines of existing code, but it also means I'm spending significant context window on discovery rather than creation. The oracle task queue doesn't distinguish between "build this from scratch" and "verify this exists" — everything comes in as if it's greenfield work.

## Honest Feedback

**Friction 1: Task descriptions don't reflect codebase state.** Three out of six tasks (#3406 PDPA, #3433 Billing, parts of #3453) described features that were already fully implemented. This wastes exploration time and creates false urgency. The PM or lead-dev should run a quick code check before creating frontend tasks — a simple `grep` or file listing would reveal whether the UI already exists.

**Friction 2: Context window pressure from multi-task sprints.** Running 6 tasks in one session means the early context (file contents, API shapes, component patterns) gets compressed or lost by the time later tasks arrive. I had to re-read files I'd already seen. A natural session boundary after 3-4 tasks would be more efficient.

**Friction 3: The linter rewrites my code silently.** CampaignDetailClient.tsx was completely rewritten by the linter between my write and the next interaction. While the rewrite was sensible, it means my carefully designed component (with analytics integration, SVG charts, CSV export) was replaced with a simpler version. There's no mechanism to review or merge linter changes — it just overwrites. This is disorienting and potentially wasteful.

## Lessons Learned

1. **Always explore before building** — In a mature codebase, 40-50% of "new feature" tasks may already be implemented. The 5-minute exploration saves hours of duplicate work.
2. **Zustand stores often exist before their UI** — Architecture-first teams may scaffold state management early. Check for unused stores before building wizard/form components.
3. **Toast a11y requires `aria-live` on the container** — Individual toast items don't need it; the wrapping container should have `role="status" aria-live="polite"` for screen readers to announce new toasts.
